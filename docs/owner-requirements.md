# LunaLighthouse Owner Requirements (Everything Needed From You)

This document is the single source of truth for what the project owner must provide to ship and maintain LunaLighthouse in phase 1.

## How To Use This Document

- Treat every section as mandatory unless explicitly marked optional.
- Complete sections in order.
- Store all sensitive values in a password manager or secret vault.
- Record completion evidence (links, screenshots, IDs, expiration dates) as you go.

## 3) Domain and DNS Configuration Requirements

You already have `lunalighthouse.app`. You still must provision and maintain these hosts.

### Required subdomains

| Host | Purpose | Required In Phase 1 |
| --- | --- | --- |
| `www.lunalighthouse.app` | Website + `.well-known` files | Yes |
| `docs.lunalighthouse.app` | Documentation | Yes |
| `builds.lunalighthouse.app` | Release artifact distribution | Yes |
| `notify.lunalighthouse.app` | Reserved for phase-2 webhook host | Yes (can be placeholder page) |

### Required DNS records

- `A`/`AAAA` or `CNAME` records for all required subdomains.
- TLS certificates active for all required subdomains.
- Certificate auto-renew enabled.
- Expiry alerting configured (email or monitoring integration).

### Required `.well-known` files on `www.lunalighthouse.app`

Deploy exactly these endpoints:

1. `https://www.lunalighthouse.app/.well-known/assetlinks.json`
2. `https://www.lunalighthouse.app/.well-known/apple-app-site-association`

Use templates in:

- `docs/templates/assetlinks.json.template`
- `docs/templates/apple-app-site-association.template`

Required substitutions:

- `<ANDROID_SHA256_FINGERPRINT>` from production Android signing key.
- `<APPLE_TEAM_ID>` from Apple Developer team.

## 4) Certificates, Signing, and Key Material

You must create, protect, rotate, and document all signing assets.

### Android signing assets (required)

| Asset | Source | Storage Requirement |
| --- | --- | --- |
| `key.jks` production keystore | Generated by `keytool` | Secret manager + offline backup |
| `key.properties` | Manual file with passwords/alias | Secret manager |
| Keystore alias and passwords | Owner-defined | Secret manager |
| SHA-256 fingerprint | Derived from `key.jks` | Documented for `assetlinks.json` |

### iOS signing assets (required)

| Asset | Source | Storage Requirement |
| --- | --- | --- |
| Apple Distribution certificate | Apple Developer/Match | Managed in Match repo |
| Provisioning profiles (`app.lunalighthouse.lunalighthouse`) | Apple Developer/Match | Managed in Match repo |
| Match encryption password | Owner-defined | Secret manager |
| Match SSH private key | Owner-generated | Secret manager |
| Match Git repo (`lunalighthouse-match`) | Owner GitHub org | Private repo + restricted access |

### Signing backup and recovery requirements

1. Keep two secure backup locations for keystore and critical credentials.
2. Maintain a recovery runbook entry for lost key scenarios.
3. Track expiration dates for Apple certificates/profiles.
4. Define who can access and rotate each secret.

## 5) GitHub Secrets Required By Workflows

These names must exist in GitHub repository secrets.

| Secret Name | Required Value |
| --- | --- |
| `KEY_JKS` | Base64-encoded Android `key.jks` |
| `KEY_PROPERTIES` | Base64-encoded Android `key.properties` |
| `APPLE_ID` | Apple ID email |
| `APPLE_ITC_TEAM_ID` | App Store Connect team identifier |
| `APPLE_TEAM_ID` | Apple Developer team identifier |
| `IOS_CODESIGNING_IDENTITY` | iOS distribution identity string |
| `MATCH_KEYCHAIN_NAME` | Fastlane keychain name |
| `MATCH_KEYCHAIN_PASSWORD` | Fastlane keychain password |
| `MATCH_PASSWORD` | Fastlane Match encryption password |
| `MATCH_SSH_PRIVATE_KEY` | Private SSH key for Match repo access |
| `MATCH_GIT_URL` | SSH URL for Match signing repo |

### Secret hygiene requirements

- No secrets in git-tracked files.
- Rotate secrets on team change, suspected leak, or annually.
- Record last rotation date and owner for each secret.
